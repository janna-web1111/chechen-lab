# Chechen Webapp Builder

## Назначение

Реализовывать MVP приложения для изучения чеченского как простое браузерное приложение: путь A0 -> A1, 5 тем, короткие уроки, карточки, квизы, прогресс и повторение.

## Порядок skill

Этот skill идет третьим:

1. `01-chechen-mvp-guardian` - сначала проверить рамки MVP.
2. `02-chechen-content-builder` - затем подготовить или проверить контент.
3. `03-chechen-webapp-builder` - потом реализовать веб-MVP в коде.

## Стандартные решения реализации

- Сначала строить онлайн веб-приложение.
- Использовать React или Next.js, если стек проекта еще не выбран.
- Хранить контент как статичный JSON или типизированные данные внутри приложения.
- Хранить прогресс в `localStorage` для MVP.
- Держать UI-тексты готовыми к локализации, русский язык - первый язык MVP.
- Отделять учебный контент от интерфейсной логики.
- Делать адаптивный интерфейс для мобильного размера, но не превращать MVP в native mobile app или PWA.

## Экраны MVP

Реализовывать только экраны, нужные для задачи, но сохранять совместимость навигации с:

- Стартовый экран.
- Путь обучения A0 -> A1.
- Экран темы.
- Урок или поток карточек.
- Экран квиза.
- Экран результата.
- Экран повторения.

Для каждого ключевого экрана предусмотреть загрузку, пустое состояние, ошибку, отсутствие прогресса и успешное состояние, если это применимо.

## Рабочий процесс разработки

1. Сначала изучить существующую структуру приложения.
2. Сопоставить запрос пользователя с самым маленьким MVP-этапом из `Spek/16-план-разработки-mvp.md`.
3. Если поведение зависит от контента или прогресса, сначала изменить модели данных.
4. Реализовать UI с короткими русскими текстами и локализационно готовыми строками.
5. Сохранять прогресс после завершения карточек, квиза и темы.
6. Проверить основной путь: старт -> путь -> тема -> карточки -> квиз -> результат -> восстановленный прогресс.

## Ограничения

- Не добавлять backend, если пользователь явно не просит изменить архитектуру MVP.
- Не собирать персональные данные в MVP.
- Не добавлять login, кабинет родителей, оплату, админку, PWA, offline mode, аудио, streak или сложную аналитику.
- Не хардкодить чеченский контент в компонентах, если его можно хранить в файле данных.
- Не показывать AI-сгенерированный чеченский контент как проверенный.

## Проверка

После изменений кода запускать доступные проверки проекта:

- Type check или build.
- Unit tests, если есть.
- Lint, если есть и он не дорогой по времени.
- Ручная проверка в браузере для пользовательского UI, если доступен dev server.
