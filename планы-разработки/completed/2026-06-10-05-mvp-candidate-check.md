- Локальный dev-сервер запущен, HTTP smoke-check вернул `200 OK`.
- Пользователь вручную проверил основной путь: `start -> topics -> first topic -> cards -> quiz -> result -> saved progress -> review`.

## Зачем этот план

После плана 04 весь текущий контент 5 MVP-тем переведен в `reviewed`. Теперь нужно проверить, что пользователь может пройти первый путь без объяснений разработчика:

`start -> topics -> first topic -> cards -> quiz -> result -> saved progress -> review`.

Этот план не добавляет новые функции. Он нужен, чтобы довести текущую онлайн web MVP-версию до состояния, которое можно показать первым реальным пользователям.

## MVP-рамки

Входит:

- проверка первого входа;
- проверка 5 тем с контентом `reviewed`;
- проверка карточек, квиза, результата, сохраненного прогресса и повторения;
- проверка desktop и мобильного размера;
- исправление только блокирующих или явно мешающих UX/технических проблем;
- обновление roadmap и завершение плана после проверок.

Не входит:

- backend;
- login или аккаунты;
- PWA/offline mode;
- аудио;
- streak;
- оплата;
- аналитика;
- новые темы;
- уровни выше A1;
- `published`-статус без отдельного решения.

## P0-задачи

### P0.1. Запустить приложение и smoke-check

- Статус: `done`

Проверить:

- `npm run dev` запускается;
- главная страница отвечает `200 OK`;
- нет стартовых runtime-ошибок.

Критерий готовности:

- локальный сайт открывается для ручной проверки.

Результат:

- `npm run dev` запустился на `http://localhost:3000`;
- `curl -I http://localhost:3000` вернул `200 OK`.

### P0.2. Проверить основной пользовательский путь

- Статус: `done`

Сценарий:

1. открыть стартовый экран;
2. перейти к пути;
3. открыть первую тему;
4. пройти все карточки;
5. пройти квиз;
6. увидеть результат;
7. вернуться к пути;
8. открыть повторение;
9. проверить сохраненный прогресс после перезагрузки;
10. проверить сброс прогресса.

Критерий готовности:

- путь проходит без объяснений разработчика и без технических ошибок.

Результат:

- автоматический браузерный сценарий не запущен, потому что `playwright` не установлен в окружении;
- ручная проверка пользователем пройдена: основной путь работает правильно.

### P0.3. Проверить контентные статусы

- Статус: `done`

Проверить:

- карточки показывают `Проверено`;
- экран повторения показывает `Проверено`;
- интерфейс не называет карточки `published`;
- нет оставшегося `TBD` в пользовательском пути;
- тема не говорит, что проверенный контент все еще ожидает проверки.

Критерий готовности:

- пользователь видит проверенный, но не опубликованный как `published`, контент.

Результат:

- данные контента проверены: 5 тем, в каждой по 8 карточек;
- `TBD` в текущих карточках нет;
- дублей русских вариантов внутри тем нет;
- raw-статус `reviewed` заменен в UI на русское `Проверено`.

### P0.4. Проверить мобильный и desktop layout

- Статус: `done`

Проверить:

- первый экран;
- список тем;
- экран темы;
- карточку;
- квиз;
- результат;
- повторение.

Критерий готовности:

- нет перекрытий, клиппинга, нечитаемых кнопок или поломки сетки на мобильном и desktop размере.

Результат:

- автоматическая визуальная проверка не запущена, потому что `playwright` не установлен в окружении;
- ручная проверка пользователем пройдена: desktop/mobile layout работает правильно.

### P0.5. Исправить найденные P0/P1 проблемы

- Статус: `done`

Правила:

- исправлять только проблемы, которые мешают пройти MVP-путь или сильно ухудшают первый опыт;
- не добавлять новые продуктовые возможности;
- не менять контент без отдельной причины.

Критерий готовности:

- все найденные P0/P1 проблемы исправлены или явно записаны как известные ограничения.

Результат:

- исправлен P1 UX-дефект: экран темы больше не показывает raw-статус `reviewed`, вместо него используется `Проверено`.

### P0.6. Финальные проверки

- Статус: `automated_checks_done`

Запустить:

```bash
npm run lint
npm run build
```

Критерий готовности:

- проверки проходят;
- план можно переносить в `completed`, коммитить, пушить и открывать PR.

Результат:

- `npm run lint` прошел;
- `npm run build` прошел вне sandbox, потому что Turbopack требует создать процесс и привязаться к локальному порту.

## Известные ограничения

- Автоматический визуальный Playwright-сценарий сейчас недоступен, если `playwright` не установлен в окружении.
- Статус `published` не используется до отдельного решения.
