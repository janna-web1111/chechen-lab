# План 07. First user feedback

## Статус

- Статус: `active`
- Ветка: `plan-first-user-feedback`
- Приложение: `app-next/`
- Цель: провести первый 10-15 минутный пользовательский тест MVP и понять, хочет ли человек продолжать изучать чеченский после первого пути.

## Зачем этот план

MVP уже содержит рабочий путь, проверенный контент, локальный прогресс, повторение и visual polish. Следующий риск не технический, а продуктовый: нужно проверить, понятен ли первый вход и есть ли желание продолжать после первой темы.

Главный сценарий:

`start -> topics -> first topic -> cards -> quiz -> result -> saved progress -> review`.

## MVP-рамки

Входит:

- сценарий первого теста на 10-15 минут;
- вопросы после теста;
- таблица фиксации наблюдений;
- классификация обратной связи;
- решение о следующем минимальном шаге после теста.

Не входит:

- новые функции в приложении;
- дизайн публичного сайта;
- backend;
- login или аккаунты;
- PWA/offline mode;
- аудио;
- streak;
- оплата;
- аналитика;
- новые уровни выше A1;
- массовый опрос до проверки первого сценария.

## Аудитория первой проверки

Минимум:

- 1 взрослый начинающий или родитель;
- если возможно, 1 ребенок диаспоры с родителем рядом.

Важно: первый тест не должен проверять все аудитории сразу. Достаточно понять, проходит ли человек первый путь без объяснений и хочет ли продолжать.

## P0-задачи

## Наблюдения первого теста

### 2026-06-10. Правильный ответ в квизе стоял первым

- Источник: ручная проверка пользователем.
- Наблюдение: в квизе правильный ответ всегда находился в верхнем левом варианте.
- Риск: пользователь может угадывать позицию, а не перевод.
- Классификация: `исправить в MVP`.
- Решение: варианты ответа должны показываться в разброс, при этом правильный ответ сохраняется через `correctCardId`.
- Статус: `fixed`.

### 2026-06-10. Порядок ответов не менялся между прохождениями

- Источник: повторная ручная проверка пользователем.
- Наблюдение: после исправления позиции правильного ответа варианты все равно стояли в одном порядке при каждом прохождении квиза.
- Риск: пользователь может запомнить расположение вариантов, а не перевод.
- Классификация: `исправить в MVP`.
- Решение: при каждом запуске квиза создавать новый seed попытки и перемешивать варианты стабильно внутри этой попытки.
- Статус: `fixed`.

### 2026-06-10. Enter не переводил к следующему вопросу

- Источник: ручная проверка пользователем.
- Наблюдение: после выбора ответа хотелось нажать Enter и перейти дальше без отдельного клика по кнопке.
- Риск: квиз ощущается медленнее и менее удобным на desktop.
- Классификация: `исправить в MVP`.
- Решение: после выбранного ответа Enter запускает действие `Следующий вопрос` или `Показать результат`.
- Статус: `fixed`.

### 2026-06-10. Нужен стиль как в прежнем kids-прототипе

- Источник: просьба пользователя и референс `https://janna-web1111.github.io/chechen-kids-app/`.
- Наблюдение: текущий MVP был аккуратным, но менее похожим на детское учебное приложение из прототипа.
- Риск: первый экран и карточки могут восприниматься слишком взросло для проверки с детьми и родителями.
- Классификация: `исправить в MVP`.
- Решение: перенести визуальные принципы прототипа без расширения функций: Fredoka, теплый фон с точками, округлые карточки, кораллово-розовые и желтые акценты, игровые кнопки с нижней тенью.
- Статус: `fixed`.

### 2026-06-10. Грамматическая заметка мешала экрану темы

- Источник: ручная проверка пользователем.
- Наблюдение: короткая грамматическая заметка внизу темы воспринималась как лишний блок внутри учебного потока.
- Риск: пользователь отвлекается от карточек и квиза.
- Классификация: `исправить в MVP`.
- Решение: убрать inline-блок `Короткая заметка` из темы и вынести такие объяснения на отдельный экран `Правила грамматики`.
- Статус: `fixed`.

### P0.1. Подготовить сценарий теста

- Статус: `todo`

Сценарий для пользователя:

1. Открыть приложение.
2. Самостоятельно понять, с чего начать.
3. Перейти к списку тем.
4. Открыть первую тему.
5. Пройти карточки первой темы.
6. Пройти квиз.
7. Посмотреть результат.
8. Вернуться к повторению.
9. Обновить страницу и проверить, что прогресс сохранился.

Критерий готовности:

- тест можно провести без дополнительных объяснений разработчика.

### P0.2. Подготовить вопросы после теста

- Статус: `todo`

Вопросы:

1. Было ли понятно, с чего начать?
2. Было ли понятно, что это путь A0 -> A1?
3. Было ли интересно проходить карточки?
4. Был ли квиз понятным?
5. Хотелось бы продолжить после первой темы?
6. Что мешало больше всего?
7. Какие слова, переводы или подсказки вызвали сомнения?
8. Удобно ли было на телефоне или в браузере?
9. Что было бы самым полезным следующим улучшением?

Критерий готовности:

- вопросы проверяют интерес и понятность, а не собирают список будущих функций.

- Статус: `todo`

Для каждого теста фиксировать:

| Поле | Значение |
|------|----------|
| Дата | |
| Участник | взрослый / родитель / ребенок с родителем / другое |

| Дошел до карточек | да / нет |
| Дошел до квиза | да / нет |
| Завершил первую тему | да / нет |
| Результат квиза | |
| Понял прогресс | да / нет / частично |
| Захотел продолжить | да / нет / неясно |
| Где остановился | |
| Прямые цитаты | |
| Замечания по контенту | |
| Замечания по интерфейсу | |
| Идеи новых функций | |

Критерий готовности:

- после теста можно отделить поведение пользователя от мнений и пожеланий.

### P0.4. Классифицировать обратную связь

- Статус: `todo`

Классификация:

- `исправить в MVP` - ломает первый вход, обучение, квиз, прогресс или доверие к контенту;
- `проверить еще` - важный сигнал, но данных мало;
- `после MVP` - полезно, но не нужно для проверки первых 10 минут;
- `отклонить сейчас` - раздувает продукт или противоречит MVP.

Критерий готовности:

- каждая проблема имеет приоритет и не превращается автоматически в новую большую функцию.

### P0.5. Выбрать следующий шаг

- Статус: `todo`

После 1-2 тестов выбрать одно:

- исправить UX первого входа;
- исправить карточки/квиз/прогресс;
- отправить спорный чеченский контент на дополнительную проверку;
- готовить минимальный публичный сайт/дизайн;
- провести еще один тест на другой аудитории.

Критерий готовности:

- выбран один следующий план, а не широкий backlog.

## Критерий завершения плана

План можно завершить, когда:

- сценарий теста подготовлен;
- вопросы подготовлены;
- шаблон наблюдений подготовлен;
- проведен минимум 1 тест или явно зафиксировано, что тест переносится;
- обратная связь классифицирована;
- выбран следующий минимальный шаг;
- roadmap обновлен;
- план перенесен в `completed`;
- branch запушен и создан PR в `main`.
