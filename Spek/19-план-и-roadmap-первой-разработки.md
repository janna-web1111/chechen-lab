# План и roadmap первой разработки

## 1. Назначение документа

Этот документ описывает практический порядок первой разработки MVP: с чего начать, что проще всего проверить и как двигаться маленькими проверяемыми шагами.

## 2. Главный принцип первой разработки

Начинать нужно не с полной платформы, а с самого короткого проверяемого пути:

**старт -> список тем -> первая тема -> карточки -> квиз -> результат -> сохраненный прогресс.**

Если этот путь работает, MVP уже можно показывать первому проверяющему или тестовому пользователю.

## 3. Что легче всего проверить первым

Самая простая проверка:

- пользователь открывает приложение;
- понимает, что это путь A0 -> A1;
- видит 5 тем;
- открывает первую тему;
- проходит карточки;
- проходит квиз;
- видит результат;
- после перезагрузки прогресс не теряется.

Эта проверка лучше, чем начинать с регистрации, backend, аудио, админки или сложного дизайна.

## 4. Рекомендуемая структура разработки

### Milestone 0. Рабочая основа

Цель: приложение открывается в браузере и показывает стартовый экран.

Задачи:

- создать структуру приложения;
- добавить файл контента;
- добавить файл UI-строк;
- добавить базовые стили;
- добавить простое переключение экранов.

Критерий готовности:

- можно открыть приложение и перейти со стартового экрана к пути обучения.

### Milestone 1. Контентный путь

Цель: пользователь видит 5 тем и понимает порядок обучения.

Задачи:

- показать 5 тем из данных;
- показать статус темы;
- показать общий прогресс;
- открыть экран темы;
- показать количество карточек и квиз.

Критерий готовности:

- пользователь понимает, с какой темы начать.

### Milestone 2. Карточки

Цель: пользователь проходит карточки одной темы.

Задачи:

- показать чеченское слово или `TBD`, пока контент не проверен;
- показать русский перевод;
- показать подсказку по чтению;
- добавить кнопки назад и дальше;
- сохранять изученные карточки.

Критерий готовности:

- пользователь может пройти все карточки первой темы без ошибки.

### Milestone 3. Квиз

Цель: пользователь проверяет себя после темы.

Задачи:

- показывать 5 вопросов;
- показывать 4 варианта ответа;
- выделять правильный и неправильный ответ;
- считать процент;
- показывать экран результата;
- разрешить повторное прохождение.

Критерий готовности:

- пользователь проходит квиз и видит результат в процентах.

### Milestone 4. Прогресс

Цель: прогресс сохраняется локально.

Задачи:

- сохранять изученные карточки;
- сохранять завершенные темы;
- сохранять результаты квизов;
- восстанавливать состояние после перезагрузки страницы;
- добавить сброс прогресса для тестирования.

Критерий готовности:

- после перезагрузки браузера прогресс не теряется.

### Milestone 5. Повторение

Цель: пользователь может вернуться к изученным словам.

Задачи:

- показать список изученных карточек;
- показать тему каждой карточки;
- показать перевод и подсказку;
- обработать пустое состояние, если карточек еще нет.

Критерий готовности:

- изученные слова доступны на отдельном экране повторения.

### Milestone 6. Первая ручная проверка

Цель: проверить основной путь без объяснений разработчика.

Проверить:

- новый пользователь без прогресса;
- прохождение первой темы;
- прохождение квиза;
- повторное прохождение квиза;
- сохранение прогресса;
- сброс прогресса;
- мобильный экран;
- desktop-экран;
- пустое повторение;
- повторение после изучения карточек.

Критерий готовности:

- тестовый пользователь может пройти первую тему за 10-15 минут.

## 6. Что не делать в первой разработке

До первой проверки MVP не делать:

- backend;
- регистрацию;
- личные аккаунты;
- родительский кабинет;
- оплату;
- PWA;
- native mobile app;
- offline mode;
- аудио;
- streak;
- сложную аналитику;
- сложный AI;
- уровни выше A1;
- полноценную админку.

## 7. Roadmap по версиям

### Version 0.1. Статический прототип

Цель: быстро увидеть основной путь в браузере.

Входит:

- статический frontend;
- 5 тем;
- карточки;
- квиз;
- `localStorage`;
- повторение;
- русский интерфейс.

Не входит:

- проверенный финальный чеченский контент;
- backend;
- аккаунты;
- аудио.

### Version 0.2. Контентная проверка

Цель: заменить `TBD` на проверенный чеченский материал.

Входит:

- проверка написания;
- проверка переводов;
- проверка подсказок по чтению;
- исправление дублей;
- финальные вопросы квиза.

### Version 0.3. UX-проверка

Цель: проверить, понятен ли первый опыт.

Входит:

- ручная проверка первого входа;
- проверка мобильного размера;
- улучшение текстов интерфейса;
- исправление очевидных UI-проблем;
- проверка 10-15 минутной сессии.

### Version 0.4. MVP-кандидат

Цель: подготовить версию для первых реальных пользователей.

Входит:

- стабильный основной путь;
- проверенный контент первой темы минимум;
- понятный прогресс;
- повторение;
- список известных ограничений.

### Version 1.0. MVP

Цель: проверить интерес пользователей после первых 10 минут.

Входит:

- 5 проверенных тем;
- карточки и квизы;
- локальный прогресс;
- повторение;
- ручной сбор обратной связи.

## 8. Самый логичный следующий шаг

Следующий шаг: довести `Version 0.1` до чистого состояния в репозитории.

Для этого нужно:

1. проверить текущую папку `app/`;
2. убедиться, что основной путь работает;
3. решить, оставляем ли статический frontend или переходим на React/Next;
4. закоммитить первую версию приложения отдельно от реорганизации спецификаций;
5. после этого заняться заменой `TBD` на проверенный чеченский контент.
