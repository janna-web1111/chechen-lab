# Specification Driven Development

## Назначение

Помогать вести проект через спецификации: сначала фиксировать цель, рамки, данные, функции, план и критерии проверки, затем переходить к коду.

Этот skill не заменяет продуктовые skills Chechen Lab. Он отвечает за порядок документов и связность репозитория.

## Когда использовать

Использовать, когда нужно:

- создать или обновить specs;
- проверить, хватает ли specs перед разработкой;
- разложить документы по папкам;
- обновить roadmap, backlog или план разработки;
- подготовить `AGENTS.md`;
- проверить AI-конфигурацию проекта;
- связать specs, skills и приложение в понятную структуру.

## Главный порядок работы

1. Проверить текущую структуру проекта.
2. Найти главный источник истины в `спецификации/`.
3. Определить, что уже закрыто, а что блокирует разработку.
4. Не создавать новые документы, если можно коротко обновить существующий.
5. Если документ нужен, сделать его коротким и связанным ссылками с остальными specs.
6. После изменения документов проверить markdown-ссылки.
7. Перед завершением проверить `git status`.

## Правила SDD для проекта

- Сначала фиксировать решение в specs, затем реализовывать код.
- MVP-рамки не расширять без явного решения в specs.
- Новые функции должны ссылаться на бизнес-цель, MVP-спеку или roadmap.
- Контент чеченского языка хранить отдельно от UI и помечать статус проверки.
- Любой roadmap должен разделять `MVP`, `после MVP` и `не делать сейчас`.
- Любой план разработки должен иметь критерии готовности и проверяемый следующий шаг.
- Новые планы разработки создавать в `планы-разработки/active/`.
- Перед push завершенной работы переносить файл плана из `планы-разработки/active/` в `планы-разработки/completed/`.
- После завершения плана и перед push обновлять `дорожная-карта/хронология.md` и roadmap-файлы затронутых областей.
- После реорганизации файлов обновлять ссылки в specs, skills и `AGENTS.md`.

## Минимальная структура проекта

Ожидаемая структура:

```text
AGENTS.md
дорожная-карта/
  области/
    данные.md
    ии-анализ.md
    фронтенд-и-дизайн.md
    продукт-и-спецификации.md
    языковой-контент.md
  README.md
  хронология.md
планы-разработки/
  active/
  completed/
спецификации/
  01-GlobalSpec/
  02-FunctionalMap/
  03-FeatureSpecs/
  04-TechnicalSpecs/
  05-VisualRules-Skills/
  06-UserStories/
agents/
  Skills/
app-next/
app/
codex/
```

`app-next/` - основное Next.js приложение.

`app/` - старый статический прототип или reference.

`codex/` может оставаться пустой папкой-заготовкой, если корневой `AGENTS.md` и `agents/Skills/` уже есть.

## Проверка готовности к уроку или ревью

Перед тем как считать репозиторий готовым, проверить:

- specs существуют и связаны между собой;
- specs разложены по папкам карты документов;
- есть roadmap и план первой разработки;
- есть корневой `AGENTS.md`;
- есть project skills в `agents/Skills/`;
- приложение имеет понятную папку;
- `git status` чистый;
- изменения отправлены на GitHub.

## Формат ответа

Для проверки проекта отвечать:

1. `Готово`: что уже закрывает задание.
2. `Не хватает`: что нужно добавить или исправить.
3. `Риски`: что может смутить проверяющего.
4. `Следующий шаг`: одно конкретное действие.

Для создания specs отвечать:

1. какие документы созданы или обновлены;
2. какие ссылки обновлены;
3. какие проверки выполнены;
4. что осталось сделать перед кодом.
